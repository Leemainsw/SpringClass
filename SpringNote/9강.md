## HTTP 프로토콜
header : 일종의 메타 정보 <br>
요청 메시지를 보내는데 요청 메시지가 보내진 시간, 메시지를 보낸 브라우저의 종류, 만약 body가 있으면 body의 데이터 타입, 크기


API 토큰 <= 토큰도 헤더에 붙여서 전달하는 경우가 있음
<br>
body : 실제 데이터의 내용

#### 응답 메시지의 구조
시작줄 : (http 버전) (상태코드) (상태 메시지)

## 메소드

|메소드|기능|
|------|---|---|
|GET|리소스 조회|
|POST|리소스 생성|
|PUT|(리소스의 일부 내용을 수정한다는 의미에 더 가까운) 리소스 수정 ≓ MODIFY|
|HEAD|기본적으로는 GET 메소드와 똑같은 역할을 하지만 헤더 정보만 요청 (응답 메시지에 바디 정보가 포함되지 않음)|
|OPTIONS|소에 요청 가능한 모든 메소드 정보를 요청 (Allow 헤더 정보를 추출) <br>(단, 보안상의 이유로 서버에서 해당 메소드를 이용한 요청을 거절할 수 있음)|


Q) Get, Post만 쓰는 이유는?<br>
특히, PUT, PATCH, DELETE도 전부 Post로 보내는 이유?<br>
A) 브라우저에서 GET, POST 밖에 못 보냄 (POST는 폼으로)  
<BR>
C => POST<BR>
R => GET<BR>
U => PATCH, PUT<BR>
D => DELETE<BR>

Rest API 설계할 때 중요해짐

GET메소드를 이용한 새로운 리소스 생설 및 수정 행위는 금지 <br>
이유
1. 이미 해당 작업을 하기 위한 메소드가 존재함
2. Get은 캐시 가능한 메소드 (이게 더 중요)
